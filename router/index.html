Router

To set up a router, you need a map of path to endPoints:
  1. List the endPoint functions as [...wrapperFunction, endPointFunction], and use the Router.reduceFunc to turn this into a single function.
  2. The sequence of functions in the map is outermost wrapper to innermost endPointFunction. This establishes a visual pattern in the definition map that is easy to read.
  3. Add the HTTP method (GET, POST) as the first path/route segment.

Example code

  const endPoints = {
  "GET/admin/*":
    [Main, GoogleAuth, fetchAsset],
  "GET/admin/checkLogin":
    [Main, GoogleAuth, checkLogin],
  "GET/admin/uploaded-images":
    [Main, GoogleAuth, listAllUploadedImages],
  "GET/admin/uploadImageURL":
    [Main, GoogleAuth, uploadImage],
  "GET/admin/files.json":
    [Main, GoogleAuth, ShaETag, CacheOnSha, makeFilesJson],
  "POST/admin/addFile":
    [Main, GoogleAuth, putToGithub],

  "GET/data/":
    [Main, SameSite, fetchAsset],

  "GET/auth/login":
    [Main, redirectToGoogle],
  "GET/auth/logout":
    [Main, logout],
  "GET/auth/callback":
    [Main, googleCallback],

  "GET/v/:sha/":
    [Main, CacheOnSha, getFromGithub],
};

for (let p in endPoints)
  endPoints[p] = Router.reduceFunc(endPoints[p]);
const router = new Router(endPoints);

Router Usage

The router is used in the onFetch function as follows:
  1. The router will match the route to an endPoint `{func, names, params}`, or undefined.
  2. For a route like GET/user/:user/role/:role matched against a path like /user/bob/role/boss, the params object will be {0: "bob", user: "bob", 1: "boss", role: "boss"}.

Example code

async function onFetch(req, env, ctx) {
  const endPoint = router.match(req.method + req.url.pathname);
  if (!endPoint)
    return new Response("Not found", { status: 404 });
  const { func, params } = endPoint;
  req.params = params;
  return func(req, env, ctx);
}

Router priority:

"/fixed/path" before "/:param/path" before "/fallback/*".
    Router priority:
  1. fixed endPoint "whatever/something"
  2. param endPoint "whatever/:param"
  3. fixed fallback "whatever/*"
  4. param fallback ":param/*"
Example: 
{
  "/admin": 1,
  "/admin/*": 3,
  "/admin/manifest.json": 1,
  "/admin/files/*": 3,
  "/admin/files/hello.txt": 1,
  "/user/root": 1,
  "/user/:user": 2,
  "/user/:user/something": 1,
  "/user/:user/else": 1,
  "/user/:user/*": 4,
  "/user/root/something": 1,
}
  1. "/admin" vs "/admin/*": We want the index.html of "example.com/admin" to be separate from the function that parses paths like "example.com/admin/selection/this?or=that".
  2. "/admin/*" vs "/admin/manifest.json": We want to be able to override the "/admin/" generic handler for specific paths.
  3. "/admin/*" vs "/admin/files/*" vs "/admin/files/hello.txt": More precise routes override less precise routes.
  4. "/admin/user/root" vs "/admin/user/:user": Static routes override param routes.
  5. "/admin/user/:user" vs "/admin/user/*": /admin/user/bob => /admin/user/:user
    /admin/user/bob/something => /admin/user/*
